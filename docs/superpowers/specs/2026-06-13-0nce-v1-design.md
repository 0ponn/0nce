# 0nce v1 — Org-Membership Disclosure (design)

Date: 2026-06-13
Status: approved (design); pending implementation plan
Builds on: `SPEC.md` (v0 contract). v0 section numbers are referenced as
`v0 §N`; new guest steps extend the v0 §4 pipeline.

## 0. One-paragraph summary

v1 adds **selective disclosure of one identity header** to the v0 ZK-Email
proof. On top of everything v0 proves (a domain's DKIM key signed an email,
with a replay nullifier), v1 additionally reveals the `From` or `To` email
address from inside the DKIM-signed header set, proving *"`claimed_domain`'s
key signed an email naming address `alice@claimed_domain`."* This is the
**org-membership wedge**: the basis for later proof-of-employment /
attribute-disclosure products (the §8 fast-follows and CHAP integration).

## 1. Market context (why this scope)

- The disclosure capability is the only §8 item with a standalone market
  wedge: proof-of-X-from-email (membership, employment, identity, income).
- v1 takes the **lowest-trust, highest-tractability** cut: org/employment
  membership via a header address, reusing v0's domain binding intact.
- It is the trusted-source front-end the CHAP financing platform
  (`zkip-stark`) explicitly lacks ("trusted source binding is not
  implemented"). v1 does **not** wire into CHAP yet (see §7), but its public
  output is shaped so a later commitment can feed CHAP's ABAC path.

## 2. The cryptographic statement (extends v0 §2)

> The prover possesses an email *M* and a DKIM-Signature header *H* such that
> all v0 §2 conditions (a)–(d) hold. Disclosure is **opt-in**: when
> `disclosed_header_kind` is `None` (the default), no address is revealed and
> the statement is exactly v0's. When it is `Some(From|To)`, additionally:
> (e) the named header is among the headers covered by *H*'s `h=` tag, i.e. it
>     is part of the DKIM-signed header set;
> (f) parsing that header's address per the restricted grammar (§5) yields
>     `disclosed_address`.
>
> When disclosed, the verifier additionally learns `disclosed_address` and
> decides, as policy, whether its domain aligns with `claimed_domain`.

**Privacy by default.** Disclosure names a person, so it is never forced: the
default (`--disclose none`) proves domain possession and reveals no address,
exactly as v0 did. v1 is strictly additive — it adds an opt-in capability
without removing v0's anonymity.

This statement is load-bearing. Condition (e) is the v1 analog of v0's
`d == claimed_domain`: it is what stops a malicious prover from disclosing a
forged, unsigned address.

**Alignment is a verifier policy, not a guest assertion** (refinement found
during implementation). The guest does not require `disclosed_address`'s
domain to equal `claimed_domain`: real mail is routinely signed by a provider
whose `d=` differs from the From domain (Gmail/Workspace sign `d=google.com`
over `From: you@company.com`; services sign `d=sendgrid.net`, etc.). Both
`claimed_domain` and `disclosed_address` are public outputs, so the verifier
computes alignment itself and applies its use-case policy. The org-membership
claim ("the domain vouches for this address") holds only when aligned; the
host verifier reports ALIGNED / NOT aligned on every proof. Soundness — that
the address comes from the signed set, not a forgery — is condition (e)/§4.9
and is independent of alignment.

## 3. What v1 does NOT prove (README-prominent, extends v0 non-statements)

- Does **not** prove the prover *is* `disclosed_address`, controls that
  mailbox, or currently works at `claimed_domain`. Same possession model as
  v0: anyone who came into possession of such a signed email — including a
  forwarded copy — can produce this proof.
- True identity binding ("the prover controls `disclosed_address`") requires
  **recipient binding** (v0 §8), which is out of v1 scope.
- **v1 is a capability demo, not lending-production-sound.** The v0 §6-step-2
  pubkey-trust gap is unchanged: the verifier still accepts `claimed_pubkey`
  from the prover and must confirm it out-of-band via DNS. Until that gap is
  closed (pubkey Merkle tree / public selector / DNSSEC-in-ZK), a malicious
  prover who supplies their own key can forge any address. Closing it is a
  prerequisite for any production (e.g. lending) use, tracked separately.

## 4. Public inputs / witness / public outputs (extends v0 §3, `core`)

New `PublicInputs` field:
- `disclosed_header_kind: Option<HeaderKind>` — `None` (default) discloses no
  address (v0 privacy mode); `Some(From|To)` selects the header to reveal.
  Public so the verifier knows whether/which header was disclosed.

New `PublicOutputs` field:
- `disclosed_address: Vec<u8>` — the revealed address bytes (e.g.
  `b"alice@corp.example"`), lowercased per §5.

`Witness`: no new field. The disclosed address is derived inside the guest
from the already-witnessed `email_raw` + reconstructed signed set, so there
is nothing new for the host to lie about.

(Future hook, not built in v1: an optional
`address_commitment = Poseidon(domain_separator_addr_v1, claimed_domain,
disclosed_address)` for CHAP consumption. Reserved, not emitted.)

## 5. Guest pipeline additions (extend v0 §4)

v0 §4.1–§4.8 are unchanged. Two steps are inserted after the signed header
set is constructed (v0 §4.5) and RSA-verified (v0 §4.6):

### §4.9 Locate the disclosed header **within the signed set**

- From the header set reconstructed in v0 §4.5 (the exact `h=`-covered
  instances, bottom-matched per RFC 6376 §3.5), select the entry whose name
  case-insensitively equals `disclosed_header_kind` (`From`/`To`).
- **Assert that header name appears in the `h=` tag.** If `From`/`To` is not
  covered by `h=`, the signature does not protect it; fail (no proof).
- Extraction reads only from the reconstructed signed bytes, never from
  `email_raw` directly. This defeats the header-prepend attack (an extra
  unsigned `From:` prepended above the signed one): the guest discloses the
  instance DKIM actually signed.

### §4.10 Parse address, assert domain, emit

- Parse the selected header value per the **restricted grammar** (v1 only;
  assert-and-fail otherwise, mirroring v0's relaxed/relaxed-only discipline):
  - `addr-spec` = `local-part "@" domain`.
  - Accept either a bare `addr-spec`, or `display-name "<" addr-spec ">"`.
  - `local-part`: dot-atom (ASCII letters/digits/`.!#$%&'*+/=?^_`{|}~-`); no
    quoted strings, no comments, no folding inside the address.
  - `domain`: dot-atom of LDH labels.
  - Reject group syntax, multiple addresses, anything outside this grammar.
- Lowercase the parsed `domain`; preserve `local-part` case (RFC 5321 §2.4).
- Emit `disclosed_address` (= `local@lowercased-domain`) as a public output.
- The guest makes **no** alignment assertion (see §2). The host verifier
  computes `disclosed_address.domain == claimed_domain` (case-insensitive) and
  reports ALIGNED / NOT aligned. Subdomain/relaxed alignment as an *aligned*
  verdict (e.g. `d=corp.example` vouching `a@mail.corp.example`) is a
  documented fast-follow; v1's check is exact-match.

Nullifier (v0 §4.7) is unchanged: `Poseidon(domain_separator_v0,
claimed_domain, signature)`. Disclosing different headers from the *same*
email is the same underlying event and shares one nullifier; a second proof
over the same signature is a replay regardless of disclosed header.

Guest size budget: v0 §9's "under 500 lines" is relaxed to **under 650
lines**, and the address parser is factored into its own module
(`address.rs`) so the core pipeline stays hand-readable.

## 6. Host (§5) and verifier (§6) changes

- **Host §5:** add `--disclose <from|to>` (default `from`). Host parses the
  address for operator display only; the guest re-derives and asserts, so the
  host cannot influence the proven value.
- **Verifier §6:** print and return `disclosed_address` alongside
  `claimed_domain` and the nullifier. Accept/replay logic unchanged. Re-print
  the v0 pubkey-trust warning (now load-bearing for the membership claim).

## 7. Out of scope (explicit)

- **B** — body-field regex disclosure (role/title/employee-id). Fast-follow.
- **C** — CHAP wire handoff. v1 only reserves the commitment shape; no
  cross-system integration, no recursion (RISC0/BN254 ↔ Aiur/Goldilocks).
- Recipient binding, freshness, relaxed domain alignment, Ed25519, `l=`,
  additional canonicalization modes.
- The pubkey-trust fix (v0 §6 step 2) — flagged as a production dependency,
  not built here.

## 8. Test plan (extend v0 §7)

Must-pass:
- Real email, `--disclose from` → `disclosed_address` equals the true signed
  From address; proof verifies.
- `disclosed_address` domain equals `claimed_domain`.
- `--disclose to` path on a fixture whose `To` is in `h=`.

Adversarial (load-bearing):
- **Header-prepend:** prepend an extra unsigned `From:` above the signed one →
  guest discloses the bottom-most signed instance; disclosed value is never the
  prepended header; proof still verifies.
- **Disclose a non-`h=` header:** `disclosed_header_kind` names a header not in
  `h=` → guest panics (no proof).
- **Malformed address:** header value outside the restricted grammar → panics.
- **Misalignment** (not a panic): a disclosed address whose domain ≠
  `claimed_domain` still produces a proof; the verifier reports NOT aligned.
  Covered by the `--disclose to` must-pass on `resigned_a` (To at example.org).

Fixtures: `host/tests/fixtures/gen_org.py` mints the two cases that need a real
signature over crafted content — `org_nonh_to.eml` (To not in `h=`) and
`org_malformed_from.eml` (From has no address). The header-prepend and
misalignment cases reuse `resigned_a.eml` (in-test mutation / `--disclose to`).

## 9. Definition of done (v1)

- Builds end-to-end (host + guest) with the RISC0 toolchain.
- A real prod-mode prove produces a proof that reveals the correct address and
  verifies; `disclosed_address` appears in the verifier output.
- Guest under 650 lines, address parser isolated and hand-reviewed; second
  reviewer signs off that §2 (e)/(f) are what is actually proven.
- All §8 must-pass + adversarial tests green; the header-prepend and non-`h=`
  soundness tests are the gate.
- README documents the §3 non-statements, prominently including the
  demo-not-production caveat.
- BENCHMARKS.md gets a v1 row (prove time will rise modestly over v0).
